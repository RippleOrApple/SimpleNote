# SimpleNote 滴答式 V2 二次开发设计

## 1. 文档状态

- 日期：2026-07-17
- 状态：设计已由用户逐节确认
- 目标平台：Windows、Android
- 当前基线：SimpleNote v1.0.1
- 改造路线：建立 V2 任务域，渐进迁移现有待办数据

## 2. 目标

将 SimpleNote 改造成一款具有滴答清单式信息架构和操作效率的本地优先个人效率应用，同时保留 SimpleNote 的独立品牌、现有笔记能力和无需账号的使用方式。

最终产品包含：

- 收集箱、今天、最近 7 天、全部任务等智能清单。
- 自定义清单、标签和智能筛选。
- 任务详情、一级子任务、四档优先级、开始时间、截止时间。
- 重复任务和 Windows / Android 本地系统提醒。
- 月、周、日/议程日历。
- 任务和笔记的 Markdown 正文内嵌图片。
- 独立笔记模块，并按创建日期进入日历。
- 支持自定义名称、提示语、图标、主题色和间隔计划的习惯。
- 任务与习惯统计。
- 无账号、用户主动发起的局域网手动同步。
- Windows 多栏布局和 Android 自适应移动布局。

## 3. 已确认的产品决策

### 3.1 同步与账户

- 不加入账户体系。
- 不加入云服务。
- 保留本地优先和手动局域网同步。
- 所有核心功能离线可用。

### 3.2 笔记定位

- 笔记继续作为独立模块。
- 笔记与任务不共用数据表，也不支持互相转换。
- 日历使用笔记的 `createdAt` 作为固定归档日期。
- 编辑笔记只改变 `updatedAt`，不会移动其日历归属。

### 3.3 任务与笔记图片

- 图片可以插入任务 Markdown 描述或笔记 Markdown 正文的当前光标位置。
- 用户不需要手写 Markdown；编辑器工具栏负责选择图片并插入引用。
- 支持从 Windows 文件系统、Android 相册和 Android 相机导入。
- 图片不是内容底部的独立附件列表；它们在正文中按插入位置呈现。
- 图片支持缩略显示、点击全屏预览和删除确认。
- 任务和笔记共享附件基础设施，但附件记录通过所有者类型和所有者 ID 保持隔离。

### 3.4 不包含的能力

- 不开发番茄钟。
- 不开发多人协作、任务分配、评论或共享空间。
- 不开发云同步、Web 客户端或账号体系。
- 不开发复杂的重复任务单次实例例外；修改重复规则影响未来执行。
- 不复制滴答清单的名称、图标、插画或其他品牌资产。

## 4. 视觉与导航设计

采用“自适应高还原”方向：信息层级和效率接近滴答清单，视觉资源和品牌保持 SimpleNote 独立。

### 4.1 Windows

宽屏使用四区工作台：

1. 功能栏：任务、日历、习惯、笔记、统计、设置。
2. 清单栏：智能清单、自定义清单、标签和智能筛选。
3. 内容区：当前任务列表、日历、习惯或笔记内容。
4. 详情面板：任务详情或当前模块的辅助信息。

窗口宽度不足时允许收起详情面板和清单栏，避免主要内容被压缩。

### 4.2 Android

底部导航保留五个纯图标入口：

1. 任务：`check_circle_rounded` 风格。
2. 日历：`calendar_month_rounded` 风格。
3. 习惯：`local_fire_department_rounded` 风格。
4. 笔记：`article_rounded` 风格。
5. 更多：`grid_view_rounded` 风格，承载统计、设置与同步。

底栏不常驻显示文字。图标采用 24dp、圆头描边和统一视觉重量；选中项使用主题色与约 40×36dp 的圆角胶囊底色。每个入口保留读屏语义标签、至少 48×48dp 的点击区域和长按名称提示。

### 4.3 通用交互

- 所有任务列表顶部提供单行快速添加。
- 快速创建后可以进入详情补充日期、提醒、图片和子任务。
- 删除任务、图片、清单和习惯需要确认。
- 保存、提醒、同步和错误状态均提供明确反馈。
- 主题系统继续生效，但控件、色阶和图标统一到新的设计语言。

## 5. 功能设计

### 5.1 智能清单

- 收集箱：未指定开始时间和截止时间的未完成任务。
- 今天：所有逾期任务与今天到期的任务。
- 最近 7 天：从今天开始的连续七个自然日内到期的任务。
- 全部：所有未软删除任务，可切换显示已完成任务。

智能清单是查询结果，不单独复制任务记录。

### 5.2 自定义清单与标签

- 清单包含名称、颜色、圆润图标、排序位置和归档状态。
- 每个任务最多属于一个自定义清单。
- 一个任务可以拥有多个任务标签。
- 任务标签可创建、重命名、改色和软删除。
- 任务标签与现有笔记标签使用不同的数据表和 Repository；删除任务标签不会影响笔记标签。
- 智能筛选可组合清单、标签、日期范围、完成状态和优先级，并保存排序方式。

### 5.3 任务

任务包含：

- 标题。
- Markdown 描述。
- 未完成或已完成状态。
- 无、低、中、高四档优先级。
- 可选开始时间和截止时间。
- 全天任务标记。
- 可选重复规则。
- 一个所属清单。
- 多个标签。
- 一级子任务。
- 多个本地提醒。
- 正文内嵌图片。
- 创建、更新、完成和软删除时间。
- 设备 ID 与同步版本。

子任务只支持一级嵌套，防止递归层级导致移动端操作和排序复杂化。

### 5.4 搜索与排序

- 任务搜索覆盖标题、Markdown 纯文本、清单名称和标签名称。
- 笔记继续使用现有标题与正文搜索。
- 任务列表支持按手动顺序、截止时间、优先级和创建时间排序。
- 搜索结果点击后直接打开任务或笔记详情。

### 5.5 Markdown 与正文图片

任务描述和笔记正文都采用 Markdown 作为持久化格式，并使用同一套可视化工具栏。至少支持：

- 标题、粗体、斜体、列表、任务列表、引用、行内代码、代码块和链接。
- 从文件、相册或相机插入图片。
- 编辑模式与预览模式。

正文引用格式为：

```markdown
![首页原型](attachment://att-7f3a)
```

`attachment://` 后是稳定附件 ID。数据库和 Markdown 不保存设备绝对路径。

图片支持 JPEG、PNG 和 WebP，单张原图上限为 20 MB。导入流程为：

1. 写入附件临时目录。
2. 验证文件类型和大小。
3. 读取宽高并计算 SHA-256。
4. 生成缩略图。
5. 原子移动到正式附件目录。
6. 写入带有内容所有者的附件记录，并向对应 Markdown 插入引用。

任一步失败都不会写入 Markdown 引用或正式附件记录。

### 5.6 重复任务

支持：

- 每天。
- 每个工作日。
- 每周及指定星期。
- 每月。
- 每年。
- 自定义日、周或月间隔。
- 可选结束日期或执行次数。

重复任务使用一个稳定任务 ID。完成一次重复任务时：

1. 写入一条完成事件。
2. 根据规则计算下一次日期。
3. 将任务恢复为未完成并推进到下一次日期。
4. 保留完成事件用于历史和统计。

修改重复规则默认影响未来执行，不支持“只修改本次”或为单次实例建立例外副本。

### 5.7 系统提醒

- 提醒可以是绝对时间，也可以是相对截止时间的偏移量。
- 提醒定义参与局域网同步。
- Windows 与 Android 各自为本机注册系统通知。
- 平台生成的通知 ID 不参与同步。
- 应用启动、任务变更和同步完成后重新核对待调度通知。
- 通知权限被拒绝时任务仍正常保存，提醒显示“未在本机启用”，并提供系统设置入口。

### 5.8 日历

提供：

- 月视图。
- 周视图。
- 日/议程视图。

日历聚合：

- 按开始时间或截止时间显示任务。
- 按 `createdAt` 显示笔记。
- 显示当天计划的习惯及完成状态。

三类内容通过图标和颜色区分。点击笔记打开独立笔记编辑页；点击任务打开任务详情；点击习惯进入打卡详情。

Windows 支持在日历中拖动任务改期。Android 不提供拖动改期，使用日期面板修改，降低触屏误操作风险。笔记和习惯不允许通过拖动改变日期。

### 5.9 习惯

习惯统一为单次完成型打卡，不设置数量或时长目标。用户可以创建任意自定义习惯，并设置：

- 名称。
- 最多 60 个 Unicode 字符的自定义提示语。
- 从内置图标库中选择的图标。
- 七个预设主题色或自定义主题色。
- 开始日期。
- 间隔计划。
- 可选提醒时间。
- 归档状态。

间隔计划支持：

- 每天。
- 每 N 天，`N` 为 2 至 365。
- 每周。
- 每 N 周，`N` 为 2 至 52。
- 每月。

计划以开始日期为锚点。每周固定在开始日期对应的星期出现；每 N 天或每 N 周按完整间隔推进；每月固定在开始日期对应的日号，如果目标月份没有该日号，则使用该月最后一天。

每个计划周期只有“未打卡”和“已打卡”两种状态。同一习惯、同一计划日期最多存在一条打卡记录。连续记录按连续完成的计划周期计算；错过任一应出现周期即中断。用户可以在历史日历中对已到期的计划日期补卡或取消打卡。

- 草绿 `#78A65A`

- 珊瑚 `#CA806E`

用户也可以通过两种方式指定当前习惯的自定义主题色：

- 输入 `#RRGGBB`、`RRGGBB` 或 `R,G,B`。
- 选择一张本地图片，在设备上提取五个候选色，再选择其中一个。

统计提供周、月、年三个时间范围：

- 完成任务数量。
- 逾期任务数量。
- 按清单分布的完成任务。
- 按优先级分布的完成任务。
- 习惯完成率。
- 当前和最长连续计划周期。
- 每日趋势图。

统计从任务完成事件和习惯打卡记录查询，不引入含义不清晰的综合效率分数。

## 6. 技术架构

### 6.1 总体方向

采用 V2 任务域渐进迁移，不重写整个应用：

- 保留现有笔记、主题、路由基础设施和数据库接入方式。
- 新建 V2 任务、日历、习惯、统计和附件模块。
- Presentation 通过 Riverpod Controller 或查询 Provider 访问业务能力。
- Controller 只组织用例，不直接执行 Drift SQL 或文件操作。
- Repository 负责数据库事务、实体映射和附件生命周期。
- 平台服务封装通知、文件选择、相册和相机能力。

### 6.2 模块边界

- Task：任务、子任务、清单、标签、筛选、完成历史和重复规则。
- Calendar：聚合查询，不拥有任务、笔记或习惯数据。
- Habit：习惯定义和打卡记录。
- Notes：复用现有领域与仓库，只适配新应用壳和日历查询。
- Statistics：只读查询任务完成事件与习惯打卡。
- Attachments：任务与笔记图片的导入、缩略图、所有者校验、解析和清理。
- Notifications：将提醒定义转换为当前平台的本地通知。
- Sync V2：元数据合并、附件清单、文件传输、校验和日志。

## 8. 数据库迁移

数据库版本按阶段升级：

- Phase 1：schema 1 → 2，加入 V2 任务、清单、任务标签、智能筛选和任务/笔记通用附件表。
- Phase 2：schema 2 → 3，加入提醒与完成事件表。
- Phase 3：schema 3 → 4，加入习惯和打卡表。
- Phase 4：schema 4 → 5，扩展同步日志和附件同步状态。

Phase 1 自动创建默认收集箱，并迁移现有 `todos`：

- 保留原 ID、标题、描述、截止时间、优先级、时间戳、软删除状态、设备 ID 和版本。
- 已完成旧待办的 `completed_at` 使用原 `updated_at`。
- 无清单的旧待办进入默认收集箱。
- 迁移后业务只读写 `tasks_v2`。
- 旧 `todos` 表在本轮开发中保留为只读迁移来源，不立即删除。

每次升级前在应用支持目录创建数据库备份。迁移失败时原数据库不被覆盖，应用显示恢复说明和备份位置。

## 9. 局域网同步 V2

### 9.1 原则

- 只在用户主动操作时开启服务和同步。
- 协议先进行版本与能力握手。
- 不允许 V1 与 V2 设备静默互相覆盖；版本不兼容时明确拒绝。
- 结构化数据继续使用软删除和最后有效变更时间合并。
- 图片文件不编码为 Base64 放入 JSON 快照。

### 9.2 同步流程

1. 交换设备信息、应用版本、协议版本和能力。
2. 交换结构化元数据快照与附件哈希清单。
3. 根据 `updatedAt`、`deletedAt`、设备 ID 和版本生成确定性的合并计划。
4. 比较 SHA-256，只传输目标设备缺少的附件。
5. 文件先进入临时目录，完成长度和哈希校验。
6. 将验证成功的临时附件原子移动到以内容哈希命名的正式路径。
7. 所有必需附件均已落盘后，在数据库事务内提交元数据合并。
8. 重新调度本地通知并刷新查询。
9. 写入包含创建、更新、删除、附件字节数和错误信息的同步日志。

任一必需附件传输、校验或正式落盘失败，本轮数据库合并不提交。数据库提交失败时，已落盘但无记录引用的内容寻址文件进入清理队列。同步可以安全重试，重复传输同一哈希不会生成重复文件。

`relative_path` 和 `thumbnail_relative_path` 是设备本地字段，不进入同步快照。远端设备根据附件 ID 和 SHA-256 生成自己的相对路径与缩略图。

### 9.3 删除与清理

- 附件删除先写软删除记录。
- Markdown 中对应引用与附件软删除在同一事务中完成。
- 物理文件在删除墓碑至少完成一次成功同步或本地保留期达到 30 天后进入清理队列。
- 清理失败不影响业务数据，下次启动继续重试。

### 9.4 阶段开发期间

Phase 1 至 Phase 3 是开发里程碑，不作为最终发布版本。由于 V1 同步无法理解 V2 数据，开发版本明确禁用旧同步入口并显示升级说明；Phase 4 完成后恢复完整同步。

## 10. 错误处理

- 编辑采用防抖自动保存，并显示“保存中、已保存、保存失败”。
- 保存失败时保留内存中的用户输入和当前编辑焦点。
- 图片导入失败不写入正文引用。
- 图片文件丢失时显示可识别占位符，不让 Markdown 页面崩溃。
- 通知权限被拒绝不阻止任务保存。
- 系统通知注册失败会记录错误并允许用户重试。
- 重复任务计算异常时不推进任务日期，并保留当前任务和完成操作前状态。
- 同步网络中断、协议不兼容、文件损坏和数据库合并失败均不覆盖原有本地数据。

## 11. 性能、可访问性与隐私

- 目标数据量：1000 条笔记、3000 条任务、100 个习惯和 5000 条打卡记录下保持流畅。
- 列表和日历按可见范围查询并使用虚拟化滚动。
- 图片列表只加载缩略图，全屏查看时加载原图。
- 搜索输入防抖，数据库字段建立与查询条件匹配的索引。
- 所有纯图标按钮提供语义标签和提示。
- 触控目标不小于 48×48dp。
- 颜色不是状态的唯一表达方式。
- 数据默认只存储在用户设备中，不连接云端。
- 局域网同步服务只在用户开启时监听，并在应用关闭或用户停止后终止。

## 14. 完成标准

只有同时满足以下条件，V2 才视为完成：

- Windows 和 Android 均可离线完成任务、笔记、日历、习惯与统计的主要流程。
- 所有旧待办、笔记、主题和设置在迁移后保留。
- 任务和笔记正文图片在两个平台均可插入、预览、删除和同步。
- 重复任务、提醒、日历日期和统计结果经过自动化测试。
- 笔记按创建日期出现在月、周和日/议程视图中。
- Android 底栏只有圆润图标，无常驻文字。
- 自定义习惯支持提示语、48 个正式图标、间隔计划、七色预设、RGB 输入和本地图片取色。
- 所有支持尺寸下，习惯提示语、统计卡与历史区域均不重叠。
- 两台设备在同一局域网内可以同步所有 V2 数据和图片。
- 同步中断、错误文件和冲突不会造成现有本地数据丢失。
- `flutter analyze` 无问题，全部自动化测试通过。
- Windows 发布构建和 Android 发布构建成功。

## 15. 实施计划拆分

本设计覆盖多个连续子系统，不生成一个超大实施计划。后续分别建立四份实施计划：

1. Phase 1：任务核心、新应用壳、迁移以及任务/笔记正文图片。
2. Phase 2：时间、重复、提醒与聚合日历。
3. Phase 3：习惯与统计。
4. Phase 4：同步 V2、附件传输、回归与发布。

每份计划都使用测试驱动步骤、明确文件路径、独立验收和阶段提交。当前仅在用户审核并批准本设计文档后编写 Phase 1 实施计划。
